# **6/23~6/30にしたこと**

## 今週やったこと

- PPO-Clip の目的関数と勾配の整理
  - PPO-Clip では、Advantage の符号に応じて、望ましい方向への過剰な方策更新を抑制する
  - $$\hat{A}_t>0$$ の場合は、良い行動を選びやすくする方向に更新される
  - $$r_t(\theta)>1+\epsilon$$ になると clip 側が採用され、そのサンプルからの勾配は 0 になる
  - $$\hat{A}_t<0$$ の場合は、悪い行動を選びにくくする方向に更新される
  - $$r_t(\theta)<1-\epsilon$$ になると同様に勾配が 0 になる

  -  その結果、それなりに良いパラメータ候補が見つかった

- 複数 seed による再現性確認
  -  見つかったパラメータで、躍度あり条件の再現性を確認した
  -   同じ条件から躍度項を外した、躍度なし条件も実行した

- post-moving 区間への躍度項の導入

# **6/7~6/15にしたこと**  
## 今週やったこと
- パラメータ自動調整の方法として、ベイズ最適化
  - ベイズ最適化は、過去に試したパラメータと評価結果をもとに、次に試すべきパラメータを効率的に選ぶ方法である
  - ランダムサーチよりも少ない試行回数で良いパラメータを見つけられる可能性がある
  - ただし、最終的には各候補パラメータでシミュレーションを実行し、結果を評価する必要がある(計算コストの問題）
  - 既存のシミュレーション結果を再利用できる（過去に実行した条件と結果を利用すれば、パラメータ探索の初期情報として使える可能性がある）
  - 一方で強化学習では乱数初期値や探索の影響により、特定の結果が偶然うまくいった可能性もある
- ベイズ最適化の代替案として、direct shooting法
    - direct shooting 法は、方策を学習するのではなく、1本の運動に対するトルク列を直接最適化する方法である
    - 具体的には、まず適当なトルク制御点列を用意し、それを補間して各時刻のトルク列を作る
    - そのトルク列を2関節アームの運動方程式に入力し、手先軌道、速度、加速度、躍度などを計算する
    - 得られた軌道に対して、目標時刻での距離、終端速度、終端加速度、躍度、post-moving中の速度などからコストを計算する
    - そのコストが小さくなるように、トルク制御点列を勾配降下法によって更新する
    - この処理を繰り返すことで、指定したコスト関数のもとで望ましいトルク列と軌道を求める
    - 乱数で初期のトルク制御点を決めるため、局所解に落ちやすいから、いくつか乱数を変えて回して最高の結果を用いる

## 基本方針
- パラメータ候補の作成は Codex が担当する
- 実行はユーザーが `run_batch.sh` で行う
- 結果の比較・評価・次候補の再提案は Codex が担当する
- パラメータ変更は json 
- 1回の比較単位は 3候補 1 batch とする

## 評価の優先順位
1. 成功率
2. ベル型速度プロファイル
3. 直線性
4. 報酬
5. 補助的に評価時エピソード長

## 各評価値の意味と出し方

### 成功率
使う値
- `success_rate_1000`
- `success_rate_100`
- `success_rate_all`

出し方
- `episode_full_metrics.csv` などのエピソード単位ログから、成功フラグ列の平均を取る
- `success_rate_all` は全エピソード平均
- `success_rate_100` は直近100エピソード平均
- `success_rate_1000` は直近1000エピソード平均

見方
- まず成功率が下がっていないかを見る
- 成功率が大きく悪化した候補は原則不採用

### ベル型速度プロファイル
使う値
- `peak_time_ratio`
- `peak_count`
- `bell_peak_error`

出し方
- `end_result_2joint_post_env.csv` などの手先軌道ログから速度時系列を使って計算する
- `peak_time_ratio = 速度ピーク時刻 / 運動時間`
- `peak_count = 局所ピーク数`
- `bell_peak_error = abs(peak_time_ratio - 0.5)`

理想
- `peak_time_ratio` は 0.5 に近い
- `peak_count` は 1
- `bell_peak_error` は 0 に近い

見方
- 成功率が同程度なら、まずここで比較する
- 単峰で、ピークが運動中盤に来る候補を優先する

### 直線性
使う値
- `path_length_ratio`

出し方
- 手先軌道から軌道長と始点終点の直線距離を求める
- `path_length_ratio = 軌道長 / 始点終点距離`

理想
- 1.0 に近いほどよい

見方
- 成功率とベル型指標が同程度なら、より 1.0 に近い候補を優先する

### 報酬
使う値
- `mean_total_reward_1000`
- `mean_total_reward_100`
- `last_eval_mean_reward`

出し方
- 学習ログの各エピソード報酬から平均を計算する
- 評価実行時の `eval_metrics.csv` から最終評価報酬を取得する

見方
- 主指標ではなく補助指標
- 他の指標が拮抗した場合に参考にする

### 評価時エピソード長
使う値
- `last_eval_mean_ep_length`

出し方
- `eval_metrics.csv` の最後の評価行から取得する

見方
- 補助指標
- 最大ステップ張り付きが続いていないかの確認に使う

- 1候補で変更してよいパラメータは最大2個
- 原則 ±20% 以内

## 2. 今週のToDo
- 論文を読む
  - Hippocampal replays under the scrutiny of reinforcement learning models
    - Romain Cazé,* X Mehdi Khamassi,* Lise Aubin, and Benoît Girard
    - Journal of neurophysiology, 2018
    - 海馬によるリプレイは記憶保持だけでなく強化学習に関与してるらしい。そこで、観測されるリプレイの型(foward など）・場所（物理的位置：報酬をもらえる場所や意思を決定する場所）・タイミング（睡眠中や覚醒中）が、どんな強化学習（MF/MB/Dyna）のどの計算過程（学習か推論か）に対応するのが自然かを整理し、実験で検証可能な予測（もし観測が「意思決定点で forward が多い」なら、単なる復習（MF）よりも、候補行動を評価する計画推論（MB/Dyna）が濃厚になる。）として提示する
    - Model-Free(MF-RL)：状態や行動の価値を、経験した報酬から直接更新
    - Model-Based（MB-RL）:遷移Tと報酬Rからなる world model を構築(頭の中で先をシミュレーションしてから行動を決める)
    - Dyna:混合（モデルフリーの学習を基本にしつつ、モデル（world model）も使って“仮想の経験”を作り、それでもう一回学習）
  - The hippocampal–striatal axis in learning, prediction and goal-directed behavior
    - Pennartz, C. M. A., et al.
    - Trends in neurosciences ,2011
    - ラットのニューロンの活動を手掛かりに海馬から線条体に対して、どんな情報が入ってくるのが妥当かを整理し、モデルの提案
    - 提案モデル：線条体を単純なActor–Criticとは捉えず、海馬によるリプレイや前頭前野からの入力に基づいて「次に何が起きるか（結果予測）」を更新し、その結果として次回以降の行動選択がより良くなる
